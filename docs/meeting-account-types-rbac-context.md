# 회의 메모 — 계정 유형 (권한·기능 설계용)

| 항목 | 내용 |
|------|------|
| 목적 | **사용자 권한 관리**, **데이터 범위(소속·테넌트)**, **기타 기능** 개발 시 동일 전제를 쓰기 위한 기록 |
| 회의 일자 | 2026-04-23 (세부 시각·참석자는 필요 시 보완) |

---

## 1. 확정 사항: 계정 유형 3종

운영·웹 포팅 맥락에서 아래 **세 가지 계정 유형**이 존재한다고 회의에서 정리되었음.

| ID | 유형 | 설명(회의 내용) |
|----|------|------------------|
| **T1** | **수퍼관리자** | 전사·전 기능에 대한 최상위 관리 권한을 가진 계정 |
| **T2** | **물류(총판) 계열** | **물류(총판)** 운영 주체 계정과, 그 **물류(총판)에 소속된 출판사** 계정을 포괄하는 유형. 상·하위(소속 관계)에 따른 권한·조회 범위가 달라질 수 있음 |
| **T3** | **독립 출판사** | 특정 물류(총판) 소속이 아닌 **독립** 출판사 계정 |

---

## 2. 설계 시 연결 포인트 (체크용)

- **RBAC / `Id_Logn`·F11~F89 대체**: 유형별로 허용 메뉴·API·행(지점·출판사·거래처) 스코프를 매핑할 때 T1 / T2(상·하위) / T3 를 1차 축으로 둠.
- **데이터 격리**: T2 내에서는 “총판 관점” vs “소속 출판사 관점” 접근 범위 구분 가능성.
- **레거시 변형**: 물류용 클라이언트와 출판 클라이언트가 나뉘어 있는 현장 전제와 정합 검토 — 메뉴 차이 참고: [`welove-ftp-tongpan-chulpan-chul-menu-diff.md`](welove-ftp-tongpan-chulpan-chul-menu-diff.md).
- **DEC / OQ**: 단일 테넌트 1차 전제(`DEC-008`)와 충돌하지 않게, **유형은 “역할·스코프 모델”** 로 두고 멀티테넌시 일반화는 [`legacy-analysis/open-questions.md`](../legacy-analysis/open-questions.md) **OQ-LOGIN-1** 후속 합의와 연동.

---

## 3. 후속로 남길 것 (미결)

- 각 유형별 **대표 계정**(개발·스테이징) 명명 규칙.
- T2 내부에서 총판 vs 소속 출판사를 **기술적으로 구분하는 키**(조직 ID, 출판사 코드, `Id_Logn` 필드 등) 확정.
- 수퍼관리자의 **생성·배포** 정책(최초 시드, 인수인계).
