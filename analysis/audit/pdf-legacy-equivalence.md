## 범위

- P1-C 출고 거래명세서: Sobo27, `Report_4_51.frf`, `Report_4_51-0..9.frf`
- P1-E 거래명세서: Sobo21, `Report_2_11.frf`, `Report_2_11-.frf`

`WeLove_FTP/Welove_인수인계` 하위에서 다음 후보를 확인했다.

- `Report_2_11*.frf`: 393건
- `Report_4_51*.frf`: 210건

대표 후보를 `debug/frf_decoder_poc.py`로 디코딩한 결과:

- Sobo21 후보 `.../출판setting/weelove/AutoUpdate/Report_2_11.frf`
  - page 약 `202.7mm x 240.6mm`, object 162, warning 0
  - 확인 라벨: `거래명세서`, `공급받는자보관용`, `공급자보관용`, `사업자등록`, `상호`, `주소`, `발행일`, `거래번호`, `거래처명`, `합계금액`, `총 부 수`, `코드`, `도서명`, `출판사`, `부수`, `정가`, `금액`, `비고`

  - page 약 `203.0mm x 249.0mm`, object 137, warning 0
  - 확인 라벨: `공급자보관용`, `공급받는자보관용`, `등록번호`, `전화번호`, `팩스번호`, `페이지`, `합계수량`, `도서코드`, `처리`, `수량`, `단가`, `비율`, `금액`, `비고`
- Sobo27로 알려진 `Report_4_51*.frf` 후보군은 일부 파일에서 `청구서 & 발송비내역서`, `계정 집계표` 등으로 디코딩되어,

따라서 Sobo21은 `.frf` 라벨 구조를 PDF 빌더에 우선 반영했고, Sobo27은 후보군 분류를 계속한다.

## 기능 동등성 체크리스트

정본 `.frf`가 없을 때의 통과 기준은 바이트/픽셀 동일이 아니라 다음 기능 동등성이다.

| 축 | P1-C Sobo27 | P1-E Sobo21 | 판정 기준 |
| --- | --- | --- | --- |
| 데이터 출처 | `get_order_detail` | `get_sales_statement_detail` | 신규 SQL 없이 기존 detail 응답 사용 |
| 헤더 | 거래처명, 출고일자, 전표번호, variant | 거래처명, 발행일, 거래번호, 거래처명, 지사, 공급자/공급받는자 보관용 | `data-legacy-id` 포함 |
| 본문 | 도서코드, 도서명, 구분, 수량, 단가, 금액 | 코드, 도서명, 구분, 부수, 정가, 금액, 비고 | DBGrid 매핑과 `.frf` 라벨 순서 일치 |
| 합계 | 수량합, 금액합 | 수량합, 금액합 | 라인 합계와 footer 값 일치 |
| 페이지 | A4 세로, 12mm 여백 | A4 세로, 12mm 여백 | `@page` CSS 유지 |
| 변형 | `base`, `v0..v9` | 기본 양식 | 변형 차이는 contract `customer_variants`에만 기록 |
| 저장 | Bearer 인증 fetch + Blob 다운로드 | Bearer 인증 fetch + Blob 다운로드 | 직접 `<a download>` 미사용 |

## 부족 자료와 후속 조치

1. `Report_4_51*` 후보군은 파일명만으로 출고 거래명세서라고 확정하면 안 된다. 라벨 디코딩 결과로 form 의미를 먼저 분류한다.
2. Sobo21은 `Report_2_11.frf`의 2단 보관용 구조와 라벨을 수동 PDF 빌더에 반영했다.
3. Sobo67 목록 자체 PDF가 필요하면 C7 범위 밖 신규 양식으로 분리하고, `Report_6_*` 계열 정본 매핑부터 작성한다.
