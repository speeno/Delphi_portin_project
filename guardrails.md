# 가드레일 문서 (L5 - Guardrails)

## 개요
AI 및 개발자가 **하면 안 되는 일**을 명시한 금지 사항 목록이다.
모든 팀원은 이 문서를 숙지하고 준수해야 하며, 위반 시 코드 리뷰에서 즉시 반려한다.

---

## 1. 데이터베이스 관련 금지 사항

| ID | 금지 사항 | 근거 |
|----|-----------|------|
| GR-DB-001 | 운영 DB 직접 변경 금지 | 운영 데이터 무결성 보호 |
| GR-DB-002 | Schema migration 자동 적용 금지 | 스키마 변경은 반드시 승인 게이트 통과 후 수동 적용 |
| GR-DB-003 | DELETE/UPDATE SQL 자동 실행 금지 | 데이터 손실 방지. 반드시 테스트 환경에서만 실행 |
| GR-DB-004 | 기존 컬럼명 임의 변경 금지 | 레거시 시스템과의 호환성 유지 |
| GR-DB-005 | 기존 테이블 구조 변경 금지 (병행 운영 기간) | 신/구 시스템이 동일 DB를 바라봄 |

## 2. 코드 관련 금지 사항

| ID | 금지 사항 | 근거 |
|----|-----------|------|
| GR-CODE-001 | 고객사별 분기 삭제 금지 | 고객사 커스터마이징 보존 필수 |
| GR-CODE-002 | 미확실한 로직은 추측 구현 금지 | TODO로 남기고 open-questions.md에 기록 |
| GR-CODE-003 | 기존 필드명 임의 변경 금지 | API 계약(Migration Contract)에 명시된 필드명 사용 |
| GR-CODE-004 | 트랜잭션 경계 임의 변경 금지 | 기존 커밋/롤백 단위를 보존해야 행위가 동일 |
| GR-CODE-005 | 평가(eval) 없이 코드 머지 금지 | 5축 평가 중 최소 Outcome eval 통과 필수 |

## 3. 장비/출력 관련 금지 사항

| ID | 금지 사항 | 근거 |
|----|-----------|------|
| GR-DEV-001 | 하드웨어 연동 코드는 stub/mock 우선 | 실 장비 없이도 테스트 가능해야 함 |
| GR-DEV-002 | 인쇄 양식 변경 시 승인 필요 | 승인 게이트 #4 (인쇄 결과 승인) 통과 필수 |
| GR-DEV-003 | ZPL/EPL 등 인쇄 명령어 직접 수정 금지 | 현업 승인 후 변경 |

## 4. 프로세스 관련 금지 사항

| ID | 금지 사항 | 근거 |
|----|-----------|------|
| GR-PROC-001 | Routing Harness 비율 일방적 변경 금지 | Go/No-Go 기준 충족 후 팀 합의 하에 변경 |
| GR-PROC-002 | 승인 게이트 건너뛰기 금지 | 각 게이트는 순차적으로 통과해야 함 |
| GR-PROC-003 | 상태 추적(L6) 파일 업데이트 없이 작업 완료 처리 금지 | 모든 작업은 trace를 남겨야 함 |
| GR-PROC-004 | Capture 데이터 없이 "동일하다" 주장 금지 | 정량적 비교 데이터로 입증 필수 |

## 위반 시 대응

1. **코드 리뷰**: 가드레일 위반 코드는 즉시 반려
2. **기록**: 위반 사항을 `legacy-analysis/decisions.md`에 기록
3. **재발 방지**: 위반 패턴을 CI/CD 자동 검사에 추가
4. **심각한 위반**: 운영 DB 변경 등 심각한 위반은 즉시 롤백 후 전체 팀에 공유
